# PHYS130B — Quantum Mechanics Lecture Notes

## Project Overview

Jupyter Book lecture notes for UCSD PHYS 130B (Quantum Mechanics), taught by Prof. Yi-Zhuang You. Textbook: David Tong, *Quantum Mechanics* (Cambridge University Press).

The notes live in `notes_src/` as `.ipynb` notebooks compiled to a static HTML site via `jupyter-book build`. The published site is served at `https://everettyou.github.io/teaching/PHYS130B/notes/`.

## Critical Rules

1. **Never use `NotebookEdit`.** It causes char-by-char corruption. Denied in `settings.local.json`.

2. **Never use `Edit` or `Write` tools on ANY file.** They trigger permission dialogs in Cowork mode that block autonomous operation. Denied in `settings.local.json`. Use Python file I/O via `Bash` for ALL file modifications.

3. **Always ensure `\n` terminators** on every non-final line in cell `"source"` arrays. Without them, Jupyter Book concatenates lines and rendering breaks.

4. **Always `Read` before writing.** Inspect the notebook's current state before modifying. Verify cell indices haven't shifted.

5. **Validate after bulk edits.** Run `skills/notebook-writer/scripts/safe_edit.py validate` to catch corruption before building.

6. **Clear `_build/` before rebuilding.** Stale cached copies in `_build/html/_sources/` and `_build/jupyter_execute/` persist across incremental builds.

## Build

From the `PHYS130B/` directory:
```bash
./build.sh
```
This clears `_build/`, builds the book, syncs to `notes/`, and patches paths for GitHub Pages.

For local preview, serve `notes/` with any static server (e.g., `python -m http.server 4000 -d notes`).

## Lecture Note Refactoring Plan

The notes are being refactored to support AI-era pedagogy. This is a multi-session effort tracked in `progress.md`. The work is organized into **workstreams**, each with specific deliverables.

### Completed Workstreams

The following workstreams were completed in previous runs:

- **Workstream 1: Prompt Review** — All 60 subsection notebooks reviewed and updated
- **Workstream 2: Homework** — All 60 subsection notebooks have `## Homework` (cell 3)
- **Workstream 3: Projects** — All 20 section parent notebooks have `## Project` (cell 2)
- **Workstream 4: Discussion Problems** — All 60 subsection notebooks have inline Discussion boxes (`:class: dropdown tip`)

### Workstream 5: Content Polish (84 notebooks)

Refactor existing lecture content to serve its new role as authoritative reference:
- Key definitions, results, and equations are visible and prominent (not hidden)
- Extended derivations and pedagogical narrative go in collapsible dropdown boxes
- Notes should be clean, inviting, structurally precise
- Remove wall-of-text; ensure every section has clear visual hierarchy
- The notes serve as a "knowledge graph skeleton" that AI agents will teach from

**Status**: Review and polish each notebook.

### Workstream 6: Validation

After each batch of changes:
- Run `safe_edit.py validate` or the inline validation script
- Verify notebook cell structure matches design.md
- Spot-check physics accuracy
- Professor proofreads rendered output after `./build.sh`

## Multi-Agent Workflow

This project uses a scheduled continuous-improvement loop. The workflow operates as follows:

### Agent Roles

1. **Manager Agent** (scheduled, runs every 10–15 min): Reads `progress.md`, checks what work is pending, dispatches tasks to specialized agents. Also incorporates professor's feedback from `feedback.md`.

2. **Review Agent**: Reads current notebooks and proposes specific improvements. Outputs proposals to `progress.md` under the relevant workstream.

3. **Design Agent**: Designs homework problems, project ideas, and discussion questions. Requires deep understanding of the physics content. Outputs draft content for the Writer Agent.

4. **Writer Agent**: Implements designs into notebooks using proper ipynb JSON format. Follows all rules in `experience.md` strictly. Always validates after writing.

5. **Science Agent**: Reviews written content for physics correctness. Checks equations, notation, conceptual accuracy. Flags errors for the Writer Agent to fix.

6. **Professor** (human): Runs `./build.sh`, proofreads rendered output, writes feedback to `feedback.md`. The Manager Agent picks up feedback and dispatches corrections.

### Progress Tracking

`progress.md` is the single source of truth. Format:

```markdown
## Workstream N: [Name]

### Completed
- [x] notebook-name.ipynb — [date] — [what was done]

### In Progress
- [ ] notebook-name.ipynb — [agent working on it] — [status]

### Pending
- [ ] notebook-name.ipynb

### Issues
- notebook-name.ipynb: [description of problem found]
```

### Feedback Loop

`feedback.md` is where the professor writes comments:

```markdown
## [Date]
### [notebook or general]
- [feedback item]
```

The Manager Agent reads this at the start of each run and creates tasks from it.

### Dispatch Rules

- Only ONE agent writes to a given notebook at a time (no parallel writes to same file)
- Always read the notebook before writing
- Always validate after writing
- Always update `progress.md` after completing work
- If a workstream has dependencies (e.g., Homework depends on Lecture Notes being finalized), respect the ordering
- Prefer depth over breadth: finish one notebook completely before moving to the next
