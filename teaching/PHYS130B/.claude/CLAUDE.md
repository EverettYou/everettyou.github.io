# PHYS130B — Quantum Mechanics Lecture Notes

## Project Overview

Jupyter Book lecture notes for UCSD PHYS 130B (Quantum Mechanics), taught by Prof. Yi-Zhuang You. Textbook: David Tong, *Quantum Mechanics* (Cambridge University Press).

The notes live in `notes_src/` as `.ipynb` notebooks compiled to a static HTML site via `jupyter-book build`. The published site is served at `https://everettyou.github.io/teaching/PHYS130B/notes/`.

## Critical Rules

1. **Never use `NotebookEdit`.** It causes char-by-char corruption. Denied in `settings.local.json`.

2. **Never use `Edit` or `Write` tools on ANY file.** They trigger permission dialogs in Cowork mode that block autonomous operation. Denied in `settings.local.json`. Use Python file I/O via `Bash` for ALL file modifications.

3. **Always ensure `\n` terminators** on every non-final line in cell `"source"` arrays.

4. **Always `Read` before writing.** Inspect the notebook's current state before modifying. Verify cell indices haven't shifted.

5. **Validate after bulk edits.** Run validation checks to catch corruption before building.

6. **Clear `_build/` before rebuilding.**

## Current Task: Lecture Note Maintenance

The lecture notes are high quality. All major workstreams (prompts, homework, projects, discussion boxes, content polish, derivation enrichment) are complete. The current task is **ongoing maintenance** — audit quality and improve where needed.

### Priority

1. **Professor feedback** (`feedback.md`) — always top priority. Address all new entries before doing anything else.
2. **Agent-initiated improvements** — use your own judgment to identify the most pressing issue and fix it.

### Maintenance Areas

#### 1. Content Consistency

- **Outline ↔ lecture alignment.** If the outline (`outline.md`) is outdated, update the outline. If the outline's storyline is incoherent, fix the outline first, then modify the lecture note to match.
- **Derivation quality.** If a derivation deviates from the notebook's storyline, or uses techniques not yet introduced at that point in the course, fix it. If a derivation is insufficient, expand it. If it's unnecessary or redundant, remove it.
- **Derivation pedagogy.** Derivations should be self-contained and accessible. If a derivation relies on a non-trivial math result, introduce that math from elementary principles first. Build up from what students know.
- **Prompts and homework.** If prompts (cell 1) or homework (cell 3) are not aligned with the current lecture content (cell 2), update them.

#### 2. Style Consistency

- **Admonition usage.** Admonition boxes should follow the conventions in `.claude/rules/content-style.md`. If the design rules are outdated or unreasonable, update the design — the goal is consistent and reasonable use of admonition boxes across all notebooks. Admonition titles should be concise (avoid titles that wrap to multiple lines). Strike a balance between what goes in a box and what stays in the main text — do not overpack boxes with content, unless the box is collapsed (`dropdown`) and contains a derivation or worked example.
- **Math notation.** Enforce the conventions in `.claude/rules/physics-conventions.md`:
  - Upright constants: `\mathrm{i}`, `\mathrm{e}`, `\mathrm{d}`
  - Operators get hats: `\hat{H}`, `\hat{\rho}`, `\hat{U}`; observables/eigenvalues do not: `E`, `p`, `m`
  - Projection operators and eigenstate labels: no hat (e.g., `\vert n \rangle`, not `\vert \hat{n} \rangle`)
  - Identity operator: `\hat{I}` consistently
  - Pauli operators: `\hat{\sigma}^x` / `\hat{X}` — never mix spin and QI notation in the same equation
  - Vectors: `\boldsymbol{r}`, kets: `\vert\psi\rangle` with `\vert`
- **Structural consistency.** All lesson notebooks should follow the same cell structure and section arrangement. Unify section titles where they are common across notebooks (e.g., `### Overview`, `### Summary`).

#### 3. Reference Consistency

- **Cross-references.** When a later lecture uses a result from an earlier one, cite it with a link (e.g., "as derived in [§3.2.2](3-2-2-schrodinger-equation)"). Add `seealso` boxes where they help students navigate related content.
- **Equation labels.** Important equations should be labeled (`(eq-topic-name)`) and cross-referenced where reused.
- **Forward/backward references.** If a concept is introduced briefly with a promise of later detail, verify the forward reference is correct and the later notebook delivers.

### Workflow: Auditor → Writer → Reviewer

Maintenance runs through a three-phase pipeline. The scheduled agent acts as manager, dispatching subagents for each phase.

#### Phase 1: Auditor

Audit at a **high level** — scan a chapter or the entire outline to spot global problems. The auditor does NOT fix anything. Instead, it produces a work order in `progress.md` broken down to individual notebooks, so writers can pick up focused tasks.

What the auditor does:
1. Read the outline, design rules, and physics conventions
2. Read through all notebooks in a chapter (or across chapters for cross-cutting issues)
3. Identify problems in content consistency, style consistency, or reference consistency
4. Write concrete, notebook-level work items into `progress.md` under "### Work Order"
5. Each work item specifies: which notebook, what the problem is, what the fix should be

#### Phase 2: Writer

Work on **one notebook at a time**. The writer reads the notebook, gets an overall picture, spots local problems, and fixes everything together in a systematic pass.

What the writer does:
1. Read the assigned notebook fully — understand its storyline, structure, and current state
2. Read the work order item(s) for this notebook from `progress.md`
3. Also spot any local issues not caught by the auditor (notation, broken refs, style)
4. Fix all issues in one coherent pass — do not make piecemeal edits
5. Validate the notebook after editing
6. Mark the work item as done in `progress.md` and note what was changed

#### Phase 3: Reviewer

Evaluate the writer's work for quality. The reviewer checks each modified notebook and documents any remaining problems as new work items.

What the reviewer checks:
- **Scientific correctness** — are equations right, are approximations justified, are claims accurate?
- **Language style** — is the prose pedagogical and concise? No padding, no jargon without explanation.
- **Content style** — is the presentation direct and simple? Does it follow design rules?
- **Alignment** — does the notebook match the outline storyline and design conventions?

If the reviewer finds problems, it writes them as new work items in `progress.md` for the next auditor/writer cycle.
